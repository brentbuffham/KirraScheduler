# 3D Playback View — Kirra Scheduler

**Date:** 2026-02-22 23:30 (updated 23:45)
**Status:** PLAN — awaiting approval

## Concept

A 3D animated playback of the blast schedule timeline, showing drill rigs and MPUs moving through a pit shell / mine surface. The user sees spatial drill plans with equipment physically positioned at their assigned blasts, transitioning day-by-day through the schedule. A pit shell provides the 3D context — the benches, ramps, and overall mine geometry.

## Prerequisites

1. **Pit shell / surface data** — imported via DXF (3DFACE, POLYFACE, LINE entities) or from a Kirra Application Project (`.kirra` / `.json`) which contains triangulated surfaces with `points[]` and `triangles[]`.
2. **Spatial blast data** — blasts need polygon boundaries (from DXF import or Kirra project) so drill holes have X/Y/Z positions on the mine bench.
3. **Three.js** — already a dependency in the Kirra App but not in KirraScheduler. Needs adding.
4. **GLTF models** — either imported by the user or bundled as generic defaults.

## Pit Shell / Mine Surface Import

### Source 1: DXF File (3DFACE / POLYFACE / LINE mesh)

DXF files from mine planning software (Surpac, Vulcan, Deswik, Micromine) commonly contain pit shell designs as:

| DXF Entity Type | Description | How we use it |
|----------------|-------------|---------------|
| **3DFACE** | Individual triangular or quad faces with XYZ vertices | Each face becomes 1-2 triangles in our mesh |
| **POLYFACE MESH** | A collection of vertices + face indices | Convert vertex table + face indices directly to `{points[], triangles[]}` |
| **LWPOLYLINE / POLYLINE** (3D) | Bench toe/crest strings at different elevations | Could triangulate between strings using Delaunay, or display as wireframe |
| **LINE** (3D) | Individual line segments at elevation | Display as wireframe; group by layer for bench identification |

**Implementation — `dxfSurfaceImport.js`:**
1. Parse the DXF ENTITIES section.
2. Collect all `3DFACE` entities → each has 3 or 4 vertices → store as triangles.
3. Collect `POLYFACE MESH` → parse vertex/face tables → store as triangles.
4. Group by DXF layer name (e.g., `PIT_SHELL`, `BENCH_410`, `RAMP_SURFACE`).
5. Return `{ name, points[], triangles[], bounds }` per layer/group.

**DXF Import UI:** Add a new drop zone or extend the existing DXF import to detect surface entities alongside blast polygons. If a DXF contains 3DFACE entities, offer the user a choice: "This DXF contains surface data — import as pit shell?"

### Source 2: Kirra Application Project (`.kirra` / `.json`)

The Kirra App already exports triangulated surfaces in its project files:

```javascript
// From kirraImport.js — currently only stores metadata
data.surfaces = [
  {
    name: "Pit_Shell_Design",
    points: [{x, y, z}, ...],        // Full vertex array
    triangles: [{a, b, c}, ...],      // Triangle index array
    meshBounds: {minX, maxX, minY, maxY, minZ, maxZ},
    gradient: "terrain"
  }
]
```

**Current problem:** `kirraImport.js` discards the actual `points[]` and `triangles[]` arrays — it only keeps metadata (counts, bounds). We need to **store the full geometry** so the 3D playback can render it.

**Fix:** Update `kirraImport.js` to store full surface geometry:
```javascript
// BEFORE (metadata only):
APP.kirraProjectSurfaces = surfArr.map(function(s) {
  return { name: s.name, pointCount: s.points.length, ... };
});

// AFTER (full geometry preserved):
APP.kirraProjectSurfaces = surfArr.map(function(s) {
  return {
    name: s.name || s.id || "Unnamed",
    points: s.points || [],           // KEEP full vertex data
    triangles: s.triangles || [],     // KEEP full triangle data
    bounds: s.meshBounds || null,
    gradient: s.gradient || "default"
  };
});
```

### Source 3: KGP Project File (round-trip)

The `.kgp` export should also include pit shell data so it round-trips:
```javascript
// In exportKGP():
data.surfaces = APP.kirraProjectSurfaces || [];
data.solids = APP.kirraProjectSolids || [];
```

And in `parseKGPProject()`:
```javascript
if (data.surfaces) APP.kirraProjectSurfaces = data.surfaces;
if (data.solids) APP.kirraProjectSolids = data.solids;
```

## Scene Layout

1. **Pit shell surface** — Triangulated mesh from DXF or Kirra project. Rendered as a semi-transparent coloured surface (terrain gradient or simple elevation colouring). Multiple surfaces supported (e.g., design pit + current pit).
2. **Blast polygons** — Extruded outlines on the bench, colour-coded by status:
   - Grey: planned
   - Blue: drilling in progress
   - Orange: loading in progress
   - Red: blast day
   - Green: completed
3. **Drill holes** — Small cylinders at each hole location (only for blasts with spatial data).
4. **Equipment** — GLTF models (or fallbacks) positioned at the active blast polygon centroid, on the correct bench elevation.
5. **Wireframe overlays** — Optional bench toe/crest strings if imported from DXF polylines.

### Coordinate System

- Uses the same local-origin approach as Kirra App (large UTM coords transformed by subtracting centroid).
- Z is elevation — **no Z transform** (per project rules).
- XY centroid computed from the union of all surface bounds + blast polygon bounds.

```
|=================================>--------------------------------------------|
```

- **Step Forward/Back** — single day increment
- **Jump to Start/End**
- **Speed slider** — 1x, 2x, 5x, 10x real-time
- **Day counter** — shows current simulation day and calendar date
- **Scrubber bar** — click/drag to jump to any day

### Animation Per Day

For each simulation day:
1. Check which blasts are drilling, loading, or blasting.
2. Position drill rigs at their assigned blast polygon centroids, at the correct bench Z.
3. Position MPUs at blasts that are in the loading phase.
4. Animate drill rig mast rotation (simple oscillation to simulate drilling).
5. Colour blast polygons based on phase.
6. Show a small progress bar above each active blast.

### Equipment Movement

When a drill moves from one blast to another (e.g., PV271-01 finishes S4_226_412_V1 drilling and moves to S4_226_411):
- Animate the model along a straight line from old position to new position.
- Could add a "travel day" visual (equipment shown on the path between blasts).

## Data Flow

```
Import Sources                        APP State
─────────────────────────────────     ─────────────────────
DXF (3DFACE, POLYFACE)          ──►  APP.kirraProjectSurfaces[]
Kirra Project (.kirra/.json)    ──►    { name, points[], triangles[], bounds }
KGP Project (.kgp)              ──►

DXF (blast polygons)            ──►  APP.blasts[]
Kirra Project (blastHoles)      ──►    { polygons[], holeTypes[], drillStart, ... }

GLTF models (.glb)              ──►  In-memory model cache

                                        │
                                        ▼
                                  PlaybackTimeline
                                  (builds daily state map)
                                        │
                                        ▼
                                  PlaybackScene
                                  ├── PitShellRenderer (surfaces)
                                  ├── BlastGeometry (polygons, holes)
                                  └── EquipmentModels (drills, MPUs)
                                        │
                                        ▼
                                  Three.js WebGL Renderer
                                  (canvas in 3D PLAYBACK tab)
```
